# Git 관리 가이드 (Git Management Guide)

> **목적**: 일관된 버전 관리 규칙을 확립하고, 실무에서 유용한 Git 활용법을 공유합니다.

**최종 업데이트**: 2025-12-05

---

## 1. 기본 원칙

### 1.1 언어
- **모든 커밋 메시지, PR, 이슈는 한국어로 작성**합니다.
- 글로벌 오픈소스 기여가 아닌 내부 프로젝트의 경우, 명확한 의사소통을 최우선으로 합니다.

### 1.2 지속적인 백업 (Continuous Backup)
> [!IMPORTANT]
> **1일 1커밋 이상을 권장합니다.**
> 작업 단위를 잘게 쪼개어 자주 커밋하면, 실수했을 때 복구하기 쉽고 코드 리뷰도 수월해집니다.

---

## 2. 커밋 메시지 규칙 (Commit Convention)

### 2.1 메시지 구조
```
태그: 제목 (50자 내외)

본문 (선택 사항, 72자마다 줄바꿈)
- 무엇을, 왜 변경했는지 설명
- 어떻게 변경했는지보다 '이유'에 집중

꼬리말 (선택 사항)
- 이슈 트래커 ID 등
```

### 2.2 태그 목록 (Type)

| 태그 | 설명 | 예시 |
|------|------|------|
| `Feat` | 새로운 기능 추가 | `Feat: 사용자 로그인 기능 구현` |
| `Fix` | 버그 수정 | `Fix: 로그인 시 NullPointerException 수정` |
| `Docs` | 문서 수정 | `Docs: README.md 설치 가이드 업데이트` |
| `Style` | 코드 포맷팅 (로직 변경 없음) | `Style: 코드 포맷팅 및 세미콜론 추가` |
| `Refactor` | 코드 리팩토링 | `Refactor: 사용자 인증 로직 개선` |
| `Test` | 테스트 코드 | `Test: 로그인 성공 케이스 테스트 추가` |
| `Chore` | 빌드, 패키지 매니저 등 | `Chore: requirements.txt 의존성 업데이트` |

### 2.3 좋은 예시
```text
Feat: 듀얼 폰트 시스템 구현

- UI 요소에는 가변폭 폰트(Proportional) 적용
- 데이터 표시 영역에는 고정폭 폰트(Fixed) 적용
- 설정 대화상자에서 폰트 변경 가능하도록 개선

Resolves: #123
```

## 3. 브랜치 전략 (Branch Strategy)

### 3.1 브랜치 구조
- **main**: 배포 가능한 안정 버전 (Production)
- **develop**: 개발 중인 최신 버전 (선택 사항)
- **feature/기능명**: 기능 개발 브랜치
- **fix/버그명**: 버그 수정 브랜치

### 3.2 작업 흐름 (Workflow)
1. `main` 브랜치에서 최신 코드 pull
2. 새 브랜치 생성 (`feature/login-ui`)
3. 작업 및 커밋
4. 원격 저장소로 push
5. Pull Request (PR) 생성 및 리뷰 요청
6. 승인 후 `main`으로 병합 (Squash & Merge 권장)

## 4. PR 및 이슈 가이드

### 4.1 Pull Request (PR)
PR 생성 시 다음 내용을 포함하여 리뷰어가 변경 사항을 쉽게 파악하도록 합니다.

```markdown
## 🔍 변경 사항
- 로그인 UI 레이아웃 구현
- 유효성 검사 로직 추가

## 📸 스크린샷 (UI 변경 시)
(이미지 첨부)

## 🧪 테스트 방법
1. 앱 실행
2. 로그인 버튼 클릭
3. ...
```

### 4.2 Issue
버그 제보나 기능 요청 시 명확하게 작성합니다.

```markdown
## 🐛 버그 설명
로그인 버튼을 눌러도 반응이 없습니다.

## 🔄 재현 방법
1. 앱 실행
2. 아이디 입력
3. 로그인 버튼 클릭

## ✅ 기대 동작
로그인이 되거나 에러 메시지가 표시되어야 합니다.
```

## 5. 실무 Git 레시피 (Cheat Sheet)

실수했을 때 당황하지 마세요! 자주 쓰이는 복구 명령어들입니다.

### 5.1 커밋 수정하기 (Amend)
방금 한 커밋의 메시지를 고치거나, 빠뜨린 파일을 추가하고 싶을 때 사용합니다.
```bash
# 1. 빠뜨린 파일 추가
git add missed_file.py

# 2. 커밋 덮어쓰기 (메시지 수정 가능)
git commit --amend
```
> **주의**: 이미 원격 저장소(Remote)에 push한 커밋은 amend 하지 마세요! (팀원들과 충돌 발생)

### 5.2 작업 임시 저장 (Stash)
작업 중인데 급하게 다른 브랜치로 이동해야 할 때 사용합니다.
```bash
# 현재 작업 임시 저장
git stash

# 다른 브랜치 다녀오기...

# 저장했던 작업 불러오기
git stash pop
```

### 5.3 커밋 취소하기 (Reset vs Revert)

**1) Reset (나만 쓰는 브랜치일 때)**
과거로 시간을 되돌립니다. 이후의 커밋 기록은 사라집니다.
```bash
# 바로 이전 커밋 취소 (작업 내용은 보존)
git reset --soft HEAD~1

# 바로 이전 커밋 취소 (작업 내용도 삭제 - 주의!)
git reset --hard HEAD~1
```

**2) Revert (이미 Push한 경우)**
특정 커밋의 변경 사항을 반대로 적용하는 **새로운 커밋**을 만듭니다. 기록이 유지되므로 안전합니다.
```bash
# 특정 커밋의 변경 사항 되돌리기
git revert <커밋해시>
```

### 5.4 로그 예쁘게 보기
```bash
git log --graph --oneline --all
```

### 5.5 파일 변경 사항 무시하기 (추적 중단 아님)
설정 파일 등을 로컬에서만 수정하고 커밋하고 싶지 않을 때:
```bash
git update-index --assume-unchanged config/settings.json

# 다시 추적하기
git update-index --no-assume-unchanged config/settings.json
```
